The loop that starts with a rejection

A reviewer session reads a pull request written by a builder session, finds a defect, and refuses to merge. What happens next is a workflow of its own — and it is not performed by the reviewer.

This repository runs three cold AI sessions that never share a context window. An architect writes the plan. A builder executes it and opens a PR. A reviewer validates that PR. When the reviewer says REQUEST CHANGES, the work returns to a fresh implementer session, which knows nothing except what is written in the pull request.

That last hop is what this document describes. It has run twice for real in this repository — c50cd93 (finding G2-F1, on PR #4) and 7c76337 (finding D9-F1, on PR #9) — and the PR #9 round exercised every branch of it, including the interesting one where the implementer refused to proceed because the reviewer's own instructions contradicted each other.

The separation that makes this work

The reviewer does not fix what it reviewed. A reviewer who patches the code it just reviewed leaves that patch unreviewed — the one change in the branch that no independent eye has seen is the change made by the person who already decided the branch was wrong. So the reviewer writes a handoff instead, and a different session executes it. When the fix lands, the reviewer re-reviews only the fix range.

Descriptive, not normative

Where this page and CONTRIBUTING.md, CLAUDE.md, or docs/gaps_analysis.md disagree, those documents win. This page records how their rules were actually carried out. Two conventions described below — the F<n>: commit prefix and the force-push license — appear in no governing document. They are conventions of the handoff comment itself, and they bind only the round that grants them.

2. Anatomy of a rejection

A rejection arrives as two documents, both comments on the pull request. Nothing else survives; the implementer session that reads them has no memory of the review.

2.1 The verdict

The verdict states an outcome, and every finding in it carries a status that distinguishes what was reproduced from what was merely asserted. In the PR #9 verdict, the single blocking finding read:

D9-F1 — CHANGELOG.md conflicts with origin/main; the PR is unmergeable as-is. … Verdict: CONFIRMED — reproduced by a test-merge in the review worktree, not taken from gh's mergeable field

It also opened with a conflict-of-interest disclosure, because on that round the reviewer had authored the PR it was reviewing:

I authored this PR. It is reviewed here anyway because the alternative — merging unreviewed because the author is the reviewer — is exactly the failure mode this document argues against. Findings below were reproduced, not asserted.

A true detail worth stating precisely

That verdict is a GitHub issue comment, not a review object: gh pr view 9 --json reviews returns an empty array. This is not sloppiness. The author and reviewer shared one GitHub identity, and the builder's PreToolUse guard denies gh pr review outright, precisely so that no session can sign off on its own work. The mechanism that made a formal approval impossible is the mechanism that was supposed to.

2.2 The implementer handoff

The handoff is written for a reader who does not exist yet. It opens with a cold-start contract, quoted here verbatim:

Cold-start contract. You are an implementation session with no prior context. Everything you need is in this comment and the documents it names. If your context is cleared mid-way, re-read this comment (gh pr view 9 --comments) and resume at the first unchecked checkpoint. Do not reconstruct state from memory, and do not trust any summary of this PR — only this comment and the repo.

The session that reviewed this PR will not fix it, by design: a reviewer who patches the code they reviewed leaves that patch unreviewed. You are the implementer. When you finish, that reviewer re-reviews only your fix commits.

Its five sections are load-bearing in this order:

SectionWhat it must containWhy the implementer cannot start without it
Coordinates Repository path, origin URL, PR number, branch, reviewed head SHA, the command to re-read the handoff, links to the verdict, the governing docs, the gate commands, environment quirks. A cold session cannot infer which branch, which head, or which build command. Getting these from chat means they die with the context.
Required findings Per finding: What is wrong · Reproduce it first (a runnable block plus the reviewer's observed output) · Fix shapes considered · Acceptance criteria. The reproduce block is the red. The acceptance criteria are what "done" means, and they must be checkable by command, not by opinion.
Checkpoints An ordered, tickable C0–C7 list. It is the only state that survives a context clear. See §3.
Fence Verified non-issues; optional cleanups explicitly deferred; files that must not be touched. Without it the implementer re-argues settled findings and expands the diff. See §4.
Rules Commit shape, force-push license, the HALT instruction, the STOP instruction, cleanup. These grant permissions the repo's standing rules withhold — and only for this round.

3. The C0–C7 checkpoint spine

The checkpoint list is not a courtesy. It is the implementer's only durable memory. A session's context can be cleared at any point; when it comes back, gh pr view 9 --comments must be sufficient to tell it exactly where the work stood. Chat is not a store. A summary is not a store. The PR is the store.

The spine, as issued on PR #9

#CheckpointWhat proves it
C0Read CONTRIBUTING.md; confirm branch and head SHA.The branch name and git log -1 --format=%h.
C1Run the reproduce block and observe the failure. Paste it in a PR comment.The runner's own output, verbatim.
C2Apply a fix shape; resolve per the handoff's rules.A tree, not a claim.
C3Verify every acceptance criterion, one by one, pasting each command's output.One command per criterion.
C4The repo's gates, green.Build, test, whitespace output.
C5Commit with the prescribed message; push; confirm CI green on the new head.A SHA and a check run.
C6Post the closing comment: finding → commit SHA → the AC output that proves it.The comment URL.
C7STOP. Do not merge. Do not start any other work.Nothing. That is the point.

Two properties make the spine work. It is ordered — C1 before C2 means the red is captured before the tree changes, and no amount of confidence licenses skipping it. And it is resumable — the implementer ticks boxes in a PR comment as it goes, so a cleared context resumes at the first unchecked box rather than starting over or, worse, guessing that earlier steps were done.

4. Reading the punch list

A reviewer's findings are not a flat list. They arrive in four tiers, and the tier decides what the implementer is allowed to do about each one.

TierMeaningImplementer's obligation
Required Blocks merge. Reproduced by the reviewer. Fix it, one commit per finding, prove it with the acceptance criteria.
Optional Real but not blocking. Goes to the candidates list. Do not do it now, however tempting while you are already in the file. It will be scheduled deliberately, with its own PR and its own review.
REFUTED The reviewer investigated a suspicion and cleared it. Do not re-argue it. Re-litigating a verified non-issue costs a whole round and produces nothing.
ACCEPTED A deviation the builder disclosed, which the reviewer ruled acceptable. Leave it. It is now a decision, not a defect.

The fence is that tiering made operational — the same information, restated as prohibitions, because a list of things you may do is easier to over-read than a list of things you may not. From PR #9:

Optional cleanups — explicitly NOT in scope for this round. … Do not do these now, however tempting while you're in the file.

Do not modify docs/ai-review-workflow.html. Do not touch any file outside CHANGELOG.md.

Why the fence is generous rather than grudging

A fix round is re-reviewed as a delta. Every file the implementer touches beyond the finding widens that delta, and a widened delta is reviewed less carefully per line than a narrow one. The cleanup you slipped in "while you were there" is the change most likely to ship unexamined. Scope discipline is not bureaucracy here; it is what keeps the second look meaningful.

5. The consultation ladder

Before designing a fix — and before declaring one done — the implementer consults. Three sources exist, and they do not carry equal weight. Authority is a property of the source, not of how persuasive the answer sounds.

Primary

Tier 1 — the advisor tool

A stronger reviewer model that is handed the implementer's entire transcript: the task, every tool call, every result. It is called before committing to an approach and again before declaring the work complete. Because it sees the actual evidence rather than a summary of it, it is the default.

Advisory

Tier 2 — a peer LLM, via the spec-driven-collaboration skill

The fallback when advisor is unavailable. It is not a peer of tier 1 and must not be treated as one. The skill packages the problem into a self-contained document — verbatim constitution files, real code excerpts, ranked hypotheses, targeted questions — and writes it to tmp/collaborate-{target}-{slug}-{date}.md for pasting into an external model. It carries its own halt rule: if ambiguous or conflicts detected, HALT and use AskUserQuestion. Its answer is an opinion from a model that has read a package, not a transcript.

Binding

Tier 3 — a ruling from the plan's author or the human owner

This is the only tier that decides. An architect session's rulings on register conformance and scope, or the owner's answer to an AskUserQuestion, settle the matter.

Never silently switch

When a tier-2 opinion contradicts a tier-3 ruling, surface the conflict. Do not quietly adopt the more convincing argument. The architect's standing note on the GAP-02 round said it plainly: "If you also collect Gemini's reply, treat it as advisory: where it conflicts with a ruling here, surface the conflict — do not silently switch." Being closer to the code is not authority either.

What the ladder is for — and what it cannot do

On PR #9 the implementer reached C5 and found that the handoff's F<n>: commit rule could not be satisfied literally: the recommended fix shape was a rebase, which folds the resolution into the branch's single existing commit, leaving no separate commit to prefix. It looked like an ambiguity. The advisor read the whole transcript and answered that it was not:

The reviewer dictated the exact commit message … in the same breath as "Push (--force-with-lease if you rebased)." … There's no reading to pick: the message is given. … Reject the two-commit split outright. Splitting … fabricates a history that never happened.

That consult prevented a HALT, which is as valuable as one that causes it. Every candidate reading produced a byte-identical tree; only the commit's label was in question, and the label had been supplied. Halting there would have spent the owner's attention on something the implementer could have read more carefully.

But a consult cannot substitute for a ruling. An advisor can bless a reading; it cannot know which reading the plan's author meant. When the candidate readings produce different artifacts, no amount of advice resolves it — only the author can.

6. HALT → ruling → amendment

A HALT is not a failure. It is what the process does when the instructions are wrong.

6.1 When to halt

SituationAction
The answer is in a file you have not openedOpen it. Research, not a HALT.
Two readings, same resulting artifactDecide, say which, proceed. It does not matter.
Two readings, different artifactsHALT
The handoff names a file, field, or output that does not existHALT
Two clauses of the handoff cannot both be satisfiedHALT
An acceptance criterion needs a tool, endpoint, or path that is unavailableHALT. Never weaken the criterion to route around it.

The cost of halting badly runs in one direction: halting on something you could have looked up teaches the human that your questions are not load-bearing, and the next real halt gets waved through.

6.2 The halt that happened

At C2 on PR #9 the implementer found the handoff self-contradictory:

  • AC-3 required: "no entry claims GAP-04 or GAP-05 is open."
  • Resolution rule 2 required: "do not delete main's entries."

Both were satisfiable only if main's changelog were clean. It was not: main listed PRs #7 and #8 as open, and gh pr view reported both MERGED. The implementer could not obey rule 2 and reach AC-3. It stopped and asked, via AskUserQuestion, rather than picking whichever clause it liked better.

The payload of a good HALT is short and mechanical: the two clauses, with the evidence that puts them in conflict, the candidate readings, and each reading's consequence on disk.

6.3 The ruling, and where it goes

The owner ruled that the acceptance criterion — not the implementer — was the defect, and posted an amendment to the handoff with an explicit precedence rule:

Read this together with the handoff comment above. Where they disagree, this comment wins. If your context is cleared, gh pr view 9 --comments and read both, newest last.

AC-3 was replaced by AC-3′, which turned a judgement call into a command:

AC-3′ — This branch adds exactly one entry under ## Unreleased (open PRs): its own … main's pre-existing entries under that heading are carried through verbatim and untouched, including the stale (PR #7, open) and (PR #8, open) lines.

Check: git diff origin/main -- CHANGELOG.md shows only the addition of this branch's entry — no deletions, no rewordings of anyone else's lines.

Two properties of a good ruling

It is written back to the source. The correction lives in the PR next to the instruction it corrects, not in a chat log. A future cold session reading the thread newest-last gets the amended rule for free.

It refuses to expand the diff. The underlying defect — that Unreleased (open PRs) goes stale on every merge, because nothing in the merge flow updates it — was filed as its own item (CL-1) with its own future PR. The fix round shipped a changelog that still contains two factually wrong lines, on purpose. Laundering an unrelated correction through a docs PR is how stale-by-construction records get made, and it fuses two changes into one indivisible diff so that neither gets reviewed properly.

7. Red-first, when the red is a merge conflict

The repository's TDD rule binds behavior changes: write the failing test, run it, capture the output. A fix round inherits the spirit even when there is no unit test to write. The handoff's reproduce block is the red, and it runs before the tree changes:

$ git merge --no-commit --no-ff origin/main
Auto-merging CHANGELOG.md
CONFLICT (content): Merge conflict in CHANGELOG.md
Automatic merge failed; fix conflicts and then commit the result.   # exit 1

$ git diff --name-only --diff-filter=U
CHANGELOG.md

$ grep -c '^<<<<<<<' CHANGELOG.md
2

$ git merge --abort          # tree restored before doing anything else
A red gate is a claim, not a fact

GitHub's mergeable: CONFLICTING is a claim. The test-merge is the evidence. The distinction is not pedantry: this repository's CI once reported whitespace errors on every merge for about five hours. There were none. git diff --check HEAD~1 HEAD was running against a depth-1 shallow clone, where HEAD~1 does not resolve, and a || echo '::error::whitespace errors' wrapper relabelled fatal: ambiguous argument as a whitespace failure. Root-cause a red before you believe what it says about itself.

The symmetric error is trusting a green. A green signal proves nothing about a thing whose job is to say no. When the change is a check, guard, validator, or fallback, feed it bad input and watch it refuse — passing on good input is exactly what the broken version did too.

8. Gates, F<n> commits, and the force-push license

8.1 The gates are the repo's own

Not a command the implementer prefers. For CleanDriver, CONTRIBUTING.md names three, and every PR must be green on all of them:

dotnet build nvcleanstall/CleanDriver.csproj
dotnet test  nvcleanstall/CleanDriver.Tests/CleanDriver.Tests.csproj
git diff --check

An unknown gate is not a passed gate. If the implementer cannot name the repo's build command, it stops and finds it rather than inventing one that happens to exit 0.

8.2 One commit per finding

The handoff prescribes F1:, F2:, … one per finding, so the delta re-review can isolate each fix. It also prescribes the exact message text. When the fix shape makes a literal separate commit impossible — as a rebase does — the implementer carries the prescribed message onto the resulting commit rather than fabricating a commit boundary that never existed, and discloses the deviation.

8.3 The force-push license

Force-pushing is normally forbidden. A handoff may lift that, narrowly, and must say why. PR #9's did:

This branch is a single commit and nobody else is building on it, so a force-push is safe here — this is the one case where the "do not force-push" rule below is explicitly lifted.

Even then: --force-with-lease, never bare --force. The lease is what turns "I believe nothing changed underneath me" into a fact the remote checks.

Provenance of these two rules

Neither the F<n>: prefix nor the force-push license appears in CONTRIBUTING.md. They are conventions of the handoff comment, granted per round. Cite the handoff for them — never the governing docs.

8.4 When the change has no test surface

A docs-only fix adds no production code, so no unit test is required — the TDD rule binds behavior changes and this adds none. Say that explicitly in the PR body. Silence there is indistinguishable from having skipped the rule, and the reviewer cannot tell the difference. Then let the verification battery (well-formedness, anchor resolution, a live render, the gates) carry the evidentiary weight a test would have carried.

9. The closing report and the delta re-review

The closing comment is the implementer's return contract. It maps each finding to the commit that fixed it and to the acceptance-criterion output that proves the fix — three columns, no prose in place of evidence:

FindingCommitAcceptance criterionOutput
D9-F1 — CHANGELOG.md conflicts with origin/main 7c76337 gh pr view 9 --json mergeable --jq .mergeable MERGEABLE (was CONFLICTING)
AC-3′: addition-only changelog diff7 added lines, 0 deletions
git diff --checkclean
CI green on the new headbuild-test-smoke pass 1m19s

Alongside it: the gates' raw output, and a Deviations section that states every departure from the handoff prominently. A deviation hidden behind a green suite is a returnable offense. Writing "None." when there are none is required, because a reviewer cannot distinguish no deviations from forgot to mention deviations.

Then the implementer stops. It does not merge; the owner merges. It does not review its own fix; the reviewer re-reviews, and only the fix range — origin/main..<fixhead>. That range is exactly why the fence and the one-commit-per-finding rule exist: they are what make the range small enough to read closely.

10. Worked example: PR #9 / D9-F1, end to end

The document this page's sibling describes (docs/ai-review-workflow.html) was submitted as PR #9 at head 2191361, reviewed, and held on one finding.

StepWhat happened
VerdictREQUEST CHANGES · one finding, D9-F1, CONFIRMED by test-merge, not by gh's mergeable field. Two optional cleanups deferred to candidates. Conflict of interest disclosed.
C0–C1Implementer confirmed branch and head, then reproduced the conflict: 2 hunks, CHANGELOG.md only. git merge --abort. Red pasted to the PR.
C2 — HALTAC-3 and resolution rule 2 were unsatisfiable together, because main itself listed merged PRs #7 and #8 as open. Asked rather than guessed.
RulingOwner amended the handoff: AC-3 → AC-3′ (addition-only diff). Root cause filed separately as CL-1. Implementer resumed at C2.
C2–C3Rebased onto origin/main (5e47617). git diff origin/main -- CHANGELOG.md | grep -c '^-[^-]' → 0. grep -c '^## 2026-07-08' → 1. The reviewed HTML file byte-identical to 2191361.
C4Build 0 errors · 62/62 tests · git diff --check clean.
C52191361 → 7c76337, pushed --force-with-lease. CI green. mergeable: MERGEABLE, mergeStateStatus: CLEAN.
C6–C7Closing report posted. Stopped. Worktree removed. Not merged.

The two disclosed deviations

They are recorded here because a playbook that shows only the clean path teaches nothing.

  1. The C5 commit shape. The rebase folded the fix into the branch's only commit, so no separate F1: commit existed to create. The tip commit was amended to carry the dictated message, with the original subject preserved in the body. Disclosed as a deviation, not presented as compliance.
  2. C1 ran before the merge guard was armed. The implementer's own PreToolUse guard denies git merge — which is the handoff's reproduce command. Running the red first meant arming the guard afterwards. Stated in the PR rather than quietly reordered.
A tool defect found by using it

Posting the C1 red required gh pr comment with a body quoting git merge. The guard scans the whole command string and cannot distinguish an executed command from quoted evidence, so it denied the call. Worked around with --body-file; recorded as a real defect rather than as a quirk. A guard that fires on the word rather than on the act will eventually be disabled by someone in a hurry — which is the failure mode guards exist to prevent.

The earlier round

The same loop ran on PR #4 (GAP-02, the real-download slice): the reviewer found that Jobs.Cancel threw ObjectDisposedException on a late or duplicate cancel, and the implementer landed c50cd93 — F1: guard Jobs.Cancel against a disposed CTS on late/duplicate cancel (G2-F1) — reproduced by a failing test first, in the shape the handoff prescribed.

11. Anti-patterns

Anti-patternWhy it is worse than it looks
Guessing past a contradictory acceptance criterionYou ship whichever reading you happened to prefer, behind evidence you generated yourself. The author never learns their instruction was broken, and the next implementer hits it again.
Re-arguing a REFUTED findingThe reviewer already reproduced the negative. A round spent re-litigating it produces no commit.
Doing the optional cleanups "while you're in the file"It widens the delta the reviewer must read, and buries the fix inside unrelated churn.
Smuggling an unrelated correction into the fix rangeThe fix and the housekeeping become one indivisible diff. Neither is reviewed properly, and you author bookkeeping for merges you did not perform.
Fabricating a commit boundary to make a fix look isolatedThe history claims a sequence of work that did not happen. Isolation is already automatic: the fix range is the diff against the base.
Bare --forceSilently discards whatever landed on the remote since you last looked. --force-with-lease makes the remote check the assumption you are betting on.
A reviewer patching its own reviewThe one change nobody independent has read is the change made by the party who already formed a view.
Weakening a test or an acceptance criterion to route around a blockerThe criterion existed because something needed proving. Removing it removes the proof, not the problem.
MergingNot the implementer's call. Not the reviewer's either. The owner merges.

One commit per finding, message prefixed F1:. This lets the delta re-review isolate each fix.

Test-first: the observed red (the conflict output) is required evidence and must be pasted in a PR comment before the fix.

Do not squash other commits. Force-push only as licensed by fix shape A above, and only with --force-with-lease.

HALT and ask (AskUserQuestion, or ask the human directly in your reply) on any ambiguity, any contradiction between this comment and the repo, or any missing prerequisite. Do not guess. A halt is the process working, not a failure.
