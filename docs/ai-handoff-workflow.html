Executing a plan written by another AI

In this repository an architect session writes the plan and a cold builder session executes it. The two never share a context window. Everything that survives the gap between them is a document: a register (docs/gaps_analysis.md), a kickoff prompt (prompts/gap-0N-implementation-prompt.md), and a review comment on a pull request.

1. Purpose & audience

This page documents the builder side of that exchange — the loop actually followed while delivering GAP-01 through GAP-05 and the CI fix. It is a playbook, not a retrospective, but every rule below is illustrated with something that really happened, because the parts most worth writing down are the parts that were nearly gotten wrong.

The single sentence the whole workflow reduces to closes both documents. The register states it for the series it governs — verbatim, the last line of docs/gaps_analysis.md:

Return contract: one PR per gap against main, never merged; STOP for owner review between gaps; HALT via AskUserQuestion on any ambiguity.

The kickoff prompt closes with the same contract, narrowed to the one gap it carries — verbatim, the last line of prompts/gap-01-implementation-prompt.md:

Return contract: one PR against main, never merged; report PR URL and STOP; HALT via AskUserQuestion on any ambiguity.

Same contract, two scopes: the register governs the series, the prompt governs the session. Neither sentence is a paraphrase of the other, and neither is paraphrased here — a page arguing that citations should be checkable earns nothing by asking to be trusted.

Audience

A builder session (human or AI) picking up a handoff document cold. If you are authoring the handoff instead, you want the register's §2 delivery structure and the kickoff-prompt template — not this page.

2. Anatomy of an inbound handoff

A handoff prompt is not prose to be skimmed. It has a fixed six-part shape, and each part is load-bearing. Using prompts/gap-01-implementation-prompt.md as the reference specimen:

PartWhat it doesWhy the builder cannot skip it
Cold-start framingDeclares the authoritative register and the mandated read order.It names which document wins when two disagree. Without it, "the code says X, the spec says Y" has no resolution rule.
Scope disciplineWhat this slice IS; an explicit "Do NOT implement GAP-02 through GAP-06"; the §4 out-of-scope fence.Adjacent work always looks natural from inside the code. The fence is what stops a one-gap PR from becoming three.
Mandatory workflowNumbered 1–9: preconditions, worktree, TDD, gates, self-review, one PR, stop.It is the acceptance criteria for process, and process failures get a PR returned unreviewed.
Halt rulesWhen to stop and ask instead of deciding.See §6. This is the highest-leverage section in the document.
Technical anchorsWatchpoints: stack, run modes, single-owner seams to reuse, model shapes.Each is prefixed "verify in-repo before relying on them" — the architect is explicitly telling you it may be stale.
Provenance + return contractWho wrote it, from which commit, for whom, and how to hand it back.It tells you what "done" means. Usually: open a PR, report the URL, stop.
Anchors are hints, not facts

The register itself says of the live NVIDIA endpoint: "Verify the exact response shape live … before parsing; the parser is written against the recorded fixture, not memory." Treat every technical anchor the same way. Open the file before citing it.

3. The governing chain

Before any code is written, the documents are read in order, because they are ordered by authority. From CLAUDE.md, verbatim:

The docs chain is authoritative and ordered: specs/nvcleanstall/spec.md (scope cut, FRs, ACs, stack pin) → specs/nvcleanstall/progress.md (milestone status) → specs/nvcleanstall/parity.md (verified feature-parity table) → docs/gaps_analysis.md (the closed task register for current work — its §4 out-of-scope fence is absolute) → prompts/ (cold-session kickoff prompts). CONTRIBUTING.md binds process: one slice = one worktree (../nvcleanclone-<name>) = one branch (feat/<slug> / docs/<slug>) = one PR; never merge — the owner merges.

Two properties of this chain matter more than its contents:

  1. The register is closed. "Items not in §3 are out of scope (§4), without exception." A good idea discovered mid-implementation is not a licence to implement it — it goes in the PR body as a note for the owner.
  2. The fence is absolute. §4 is honored "without exception, however natural it feels while in the code." Some fences are permanent safety boundaries (never execute an installer, never write the live registry); a gap may lift only a named part of one, and only with the acceptance criterion that proves the lift is narrow.

4. The execution loop

The loop is deliberately front-loaded: almost all of the judgement happens before the first line of production code, because that is where a wrong turn is cheapest to undo.

5. The consultation ladder

"Consult before designing" is not one action. There are three distinct escalation paths with three different authority levels, and collapsing them is the most consequential mistake available in this workflow.

PRIMARY Tier 1 — advisor()

A stronger reviewer model that automatically receives the entire conversation transcript: the task, every tool call, every result. It takes no parameters because there is nothing to summarize for it.

When: after orientation but before substantive work (writing, editing, committing to an interpretation), when stuck, when considering a change of approach, and again when the work is believed complete — with the deliverable made durable first.

Weight: serious. Deviate only on empirical failure or primary-source contradiction. A passing self-test is not evidence the advice was wrong — it is evidence the test does not check what the advice checks. If retrieved data and the advisor disagree, do not silently pick: surface the conflict in one more call.

FALLBACK Tier 2 — /collaborate → spec-driven-collaboration

Used when advisor is unavailable. It packages the design question for an external peer LLM that has no access to the filesystem, so the document must be self-contained. Six phases: context gathering → constitution loading → code collection → analysis & template population → document generation → completion report.

Its two governing rules, verbatim from the skill:

Constitution files are THE LAW

If ambiguous or conflicts detected: HALT and use AskUserQuestion

Output lands at tmp/collaborate-{target}-{slug}-{date}.md as a ten-section package: executive summary, project context, the specific problem, real code artifacts (pasted, not paraphrased), what was tried, ranked hypotheses with confidence levels, specific questions, the proposed plan, a files reference, and a compliance checklist.

BINDING Tier 3 — architect rulings

A ruling from the architect session that authored the register. It outranks both tiers above and the builder's own judgement.

The relationship between Tier 2 and Tier 3 is stated explicitly in tmp/architect-response-gap02-design-2026-07-08.md:

These are binding rulings on register conformance and scope… If you also collect Gemini's reply, treat it as advisory: where it conflicts with a ruling here, surface the conflict — do not silently switch.

Do not flatten the ladder

A peer LLM's opinion is advisory. An architect ruling is binding. When they disagree the builder does not arbitrate — it surfaces the conflict and waits. The same rule applies to the builder's own analysis: being close to the code is not authority.

6. The HALT protocol

The kickoff prompt's halt rules, verbatim:

HALT if you need clarification on any requirement or detect any ambiguity — including any contradiction between docs/gaps_analysis.md, the specs, and the code… Do not guess, do not pick an interpretation silently, do not fill gaps with plausible content.

To HALT: stop working and use AskUserQuestion. State the ambiguity precisely (file:line for each side), your candidate interpretations, and which you would pick and why. Wait for the answer before proceeding.

Environment blockers (SDK missing, push rejected, endpoint unreachable when an AC needs it, worktree path unwritable) are HALT conditions too — report exactly what failed; never work around them by weakening or skipping tests.

What a HALT must carry

  1. The ambiguity, precisely. Quote both sides with file:line. "The register says X; the code does Y."
  2. Candidate interpretations. Concrete and mutually exclusive — ideally with a preview of the artifact each would produce.
  3. Your pick, and why. HALT is not abdication. The owner should be able to answer with one click.
  4. Nothing else in flight. Stop working. Do not implement "the likely one" while waiting.
The bar for HALTing

Not every unknown is an ambiguity. A choice with a conventional default, or a fact verifiable by opening a file, is research — do it. A HALT is for a fork where the register genuinely underdetermines the deliverable, where both readings produce different artifacts, and where guessing wrong means shipping the wrong thing behind green tests.

7. TDD contract & evidence

TDD here is a reviewable property of the commit history, not a private practice. From CONTRIBUTING.md:

No production code without a failing test that requires it. If you cannot name the test that forced a line of code, the line is out of scope.

Commit ordering proves the discipline: the test lands in the same commit as (or a commit before) the implementation it specifies — never after. A PR whose history shows implementation-then-tests gets returned without further review.

Two textures of "red"

TextureLooks likeTypical cause
Compile-levelerror CS0246: type 'DownloadArtifact' could not be foundThe test names a type or method that does not exist yet — extraction of a new seam.
Assertion-levelAssert.Null() Failure … Actual: "5ab.cd"The seam exists; its behavior is wrong. The stronger of the two — it pins semantics, not shape.

Tests that are green from the start

Some tests never fail, and that is correct: back-compat pins (an existing value must not change), safety pins (a forbidden API must remain absent), and characterization pins (locking behavior that already happens to be right). They do not drive new code, so they are exempt from red-first — but the exemption must be disclosed in the PR body, by name. Silence here looks like a fabricated red. (This disclosure rule was practice before it was law; it is now recorded as amendment A2 in the task register — docs/hardening_register.md §3 — which is the citable source.)

Existing tests are pins

"Editing one to make it pass is a design smell — each such edit must be individually disclosed and justified in the PR body." A test edited to accommodate new code is the single loudest signal that the new code broke a contract.

8. Gates & the PR contract

Three gates, run from inside the worktree, every PR, independently green:

dotnet build nvcleanstall/CleanDriver.csproj
dotnet test  nvcleanstall/CleanDriver.Tests/CleanDriver.Tests.csproj
git diff --check

The PR body is the deliverable's other half. It carries a red-then-green evidence table (one row per behavior, the captured failing output beside the passing one), the gate checklist, a live exercise proving the changed path ran against the real app rather than only unit tests, and — prominently — every deviation:

Scope deviations from the governing register/spec are disclosed prominently in the PR body, never hidden behind green tests.

Then, without exception: never merge. The owner merges every PR.

9. Return contract & punch list

Do NOT merge. Do NOT start GAP-02. Report the PR URL and STOP. Expect a punch-list review comment (read via gh pr view <n> --comments); implement required findings on the same branch, one commit per finding, F<n>:-prefixed, re-run gates, post a checklist comment mapping finding → commit → test, and STOP again.

The loop terminates on the owner, twice: once after the PR is opened, and once after the punch list is addressed. "The next gap looks easy and I have context loaded" is precisely the reasoning the stop rule exists to defeat.

10. Worked examples

GAP-02 — advisor unavailable, so the question was packaged for a peer

The GAP-02 builder session needed a design review before writing a real-download implementation, and advisor was not available. It invoked /collaborate, which produced tmp/collaborate-gemini-gap02-design-2026-07-08.md: ten sections, the repo's constitution quoted verbatim as law, the actual Jobs.cs / Api.cs / app.js source pasted in, six design decisions ranked by confidence, and pointed questions. One, verbatim:

Streaming correctness: With HttpCompletionOption.ResponseHeadersRead, is there a hidden HttpClient.Timeout interaction on the body stream I should guard against …? Recommend a robust stall-detection pattern for a long download.

In parallel, the architect session returned binding rulings that ratified the plan and tightened it. Those rulings — not the peer's advice — became the implementation:

  • A separate download HttpClient with Timeout = InfiniteTimeSpan, liveness enforced by a per-read stall timeout.
  • File.Move(part, final, overwrite: true) with the write stream disposed before the move, retried for antivirus-lock races.
  • A 2 GiB default cap; a .nvidia.com host guard; idempotent duplicate-version downloads.
  • Two required edge cases the builder had not raised: path-injection from untrusted live version strings (validate ^\d{3}\.\d{2}$ before any filesystem call), and returning the in-flight job on a duplicate request.

All of them shipped. The peer reply stayed advisory.

GAP-03 — the advisor caught a bug before it existed in a PR

GAP-03 looked like a pure refactor: extract the WMI-line parsing into a testable function. The advisor, called before implementation, pointed at MarketingVersion, which stripped dots and took the last five characters. Fed a malformed driver string it produced a plausible-looking fabrication:

"31.0.15.abcd"    → "5ab.cd"
"garbage-version" → "rsi.on"

The register's own acceptance criterion demanded a "malformed driver string" case; without the advisor's prompt it would have been written to assert the garbage. The fix (require all-ASCII-digits, else surface the raw string) became the behavior the AC actually wanted, and shipped with an assertion-level red.

Why "before" matters

Had the advisor been consulted after the tests were written, the tests would have encoded the bug and passed. Consulting before committing to an interpretation is what makes the advice actionable rather than merely correct.

GAP-05 — a real HALT, and a register amended in the same PR

The GAP-05 register entry required the driver-telemetry tweak to emit "a simulated: true field in any artifact they write." But the tweak writes no artifact of its own — it is not a .reg-emitting tweak — and the install receipt already carried a top-level simulated: true that predated the gap. The clause underdetermined the deliverable: a new note file, a field on existing artifacts, or nothing at all.

That is a HALT. Work stopped; AskUserQuestion presented the candidate artifact shapes as previews. The owner ruled: put a named driverTelemetrySimulated marker on the artifacts that already record the tweak (receipt, customized manifest, config.json) — no standalone note file, because "a standalone note file whose entire content is 'nothing really happened' is output clutter."

The ruling then did one more thing, and it is the part worth copying: it was written back into docs/gaps_analysis.md as a Resolution bullet in the same PR. The ambiguity closed in the register, not merely in a chat log — so the next session to read that entry cannot re-derive the same fork.

The CI fix — trust the gate, then verify the gate

A CI run reported git diff --check found whitespace errors on main. There were none. The step ran git diff --check HEAD~1 HEAD against a shallow (depth-1) checkout, so:

fatal: ambiguous argument 'HEAD~1': unknown revision or path not in the working tree.

Git exited non-zero, and a || { echo '::error::…whitespace errors'; exit 1; } wrapper relabelled a git error as a whitespace failure. It had false-failed on every merge since CI was introduced. The gate had never once actually run.

Lesson

A red gate is a claim, not a fact. Reproduce it locally before believing its message — and never let an error handler assert a cause it has not established. The fix made the check honest (fetch enough history, compute a real base, let git print its own file:line) and moved it earlier, onto pull requests, where it can prevent rather than annotate.

11. Anti-patterns

Anti-patternWhy it is temptingWhat to do instead
Silently picking an interpretationOne reading seems obviously intended, and asking feels slow.HALT. "Do not fill gaps with plausible content." A plausible wrong artifact behind green tests is the worst outcome available.
Letting an advisory opinion override a binding rulingThe peer LLM's answer is more detailed and arrived later.Surface the conflict; do not silently switch. Authority is a property of the source, not the argument's polish.
Batching the next gapContext is warm; the next slice is small.One gap per session, one PR, then STOP. The register says so twice.
Weakening or skipping a test to route around a blockerThe gate is failing and the blocker is "environmental."Environment blockers are HALT conditions. Report exactly what failed.
Implementing a good idea found in the codeIt is obviously an improvement and you are right there."New ideas go in the PR body as notes for the owner, never into code."
Merging your own PRIt is green and the owner is asleep.Never merge. Report the URL and stop.
Hiding a deviation behind a passing suiteThe tests are green, so the change must be fine.Disclose it prominently. Green tests are not a scope argument.
