Architect PR-Validation Workflow

How a pull request written by a cold AI builder session gets validated before the owner merges it — including the feedback loop back to that session when the PR needs more work.

Why this exists

This repository runs a delegation loop. A closed task register (docs/gaps_analysis.md) defines the work; a cold Claude Code builder session — no chat history, no context beyond its kickoff prompt — implements exactly one gap and opens exactly one PR; a standing architect session independently reviews that PR; the owner merges. Nobody else merges, ever.

register (gaps_analysis.md) │ ▼ kickoff prompt ──▶ cold builder session ──▶ PR opened, builder STOPS │ ▼ architect review (this document) │ ┌────────────────────────────┴───────────────────────┐ ▼ ▼ punch list → builder fixes verdict: "merge it" (same branch, F<n>: commits) │ │ ▼ └──▶ delta re-review ────────────────────▶ owner merges

The review stage is the one that catches what the builder cannot see about itself. It has run four times (PRs #3–#6) producing three approvals and one request-for-changes. That single finding — a crash on a race the builder's own green test suite never exercised — is the whole justification for the stage, and it was only found because the reviewer reproduced it with a failing test instead of reading the diff and reasoning about it.

The two invariants

Invariant 1

The reviewer never merges. Not "usually doesn't" — never. The architect posts a verdict; the owner decides. This keeps a single human accountable for what enters main, and it is enforced server-side by branch protection.

Invariant 2

A builder's self-report is a claim, not a fact. "All gates green locally" is a hypothesis about a machine you did not observe. Every claim that matters gets re-run by the reviewer, on the reviewer's own checkout, before it appears in a verdict.

A third rule governs when the verdict is delivered: review before relay. The owner merges fast, so any message resembling "the PR is ready" is a merge trigger. The independent review therefore completes before the PR is reported with anything that sounds like approval.

Stage 1 · CI triage first

Before spending a single minute reading the diff, check the build:

gh pr checks 6
# build-test-smoke   pass   1m20s   https://github.com/.../runs/...

If CI is red, root-cause it yourself and stop there — its cause decides whether anything else in the review even matters. The builder's "green locally" does not settle it: local and CI environments differ in parallelism, resources, and available hardware.

The compounding effect

Before GAP-01 there was no test project, so CI could only build. GAP-01's first commit scaffolded CleanDriver.Tests, and from that moment dotnet test became a hard CI gate. By PR #6, "53 tests pass" is a machine-verified fact about a specific commit SHA, not a claim in a summary. Each wave's infrastructure makes the next wave's review cheaper.

Stage 2 · Isolated worktree at the PR head

git fetch origin feat/gap-02-real-download
git worktree add ../nvcleanclone-review4 origin/feat/gap-02-real-download
git -C ../nvcleanclone-review4 diff --stat origin/main...HEAD

The reviewed range is always origin/main...HEAD — merge-base to head, so an out-of-date branch doesn't smear unrelated commits into the diff.

Isolation buys two things. The working checkout stays untouched (no stashing, no accidental cross-contamination of the builder's branch), and — critically — the reviewer gets a scratch space to write throwaway repro tests in, which is what Stage 5 depends on.

Always clean up

git worktree remove ../nvcleanclone-review4 --force when the verdict is posted. A leftover worktree is side-effect residue on a shared environment; the same hygiene rule the builders are held to applies to the reviewer.

Stage 3 · Re-run the gates, then exercise it live

Inside the review worktree, run the project's own gates (defined in CONTRIBUTING.md) rather than trusting the PR body:

dotnet build nvcleanstall/CleanDriver.csproj -c Release
dotnet test  nvcleanstall/CleanDriver.Tests/CleanDriver.Tests.csproj
git diff --check origin/main...HEAD

Then drive the changed path in the running application. Unit tests prove the units; only a live run proves the wiring. This app makes that cheap by design: --headless starts the server without the WebView2 window, so every behavior is reachable from curl, and the UI is reachable from Playwright.

What that looked like in practice

  • PR #3 — ran the default path on the host's real RTX 5070: /api/catalog returned source: live with 7 genuine NVIDIA releases; then --mock-catalog returned exactly the 5 sample releases. Both halves of the seam, observed.
  • PR #4 — verified the post-download hand-off the review notes predicted: /api/package for live version 610.74 returned the relabeled sample manifest with sampleComponents: true.
  • PR #5 — a back-compat check: /api/system still returned 591.86 / isSimulated:false with a byte-identical DetectedVia string after the parsing refactor.

Stage 4 · Finder pass, and the trap to hunt first

Read the diff along several independent angles rather than in one linear pass. Scale the number of angles to the wave's shape; a backend-only wave needs fewer than a full-stack one:

  • line-by-line correctness of new code
  • regression hunt against the pre-image (what changed or disappeared)
  • cross-file consistency: contracts ↔ implementations ↔ consumers
  • reuse — does new code fork a single-owner seam instead of routing through it?
  • architecture and fence conformance (the register's §4 out-of-scope list)
  • AC and test-quality audit
The recurring trap — check this every single wave

Acceptance criteria proven against test-only variants instead of production wiring. Machinery that is shape without function: a type, field, or handler that exists and is tested but is never reached by the default production path. The question to ask of every AC is not "is there a test?" but "does that test exercise the path production actually runs?"

What passing this check looks like

  • PR #3: the wiring test calls CatalogEndpoint.Build — the exact function Api.cs serves from /api/catalog, not a re-implementation with the same shape.
  • PR #4: the fallback tests assert handler.LastRequest == null — proving no HTTP was attempted, rather than merely that mock data came back (which a broken implementation could also produce).
  • PR #6: the receipt tests read the receipt JSON off disk after the job completes, instead of inspecting an in-memory object the writer never serialized.

Stage 5 · Adversarial verify — a finding is a claim

Symmetry with Invariant 2: if the builder's report is a claim to verify, so is the reviewer's suspicion. Every candidate finding must reach one of three states before it may appear in a verdict:

VerdictRequirement
CONFIRMEDReproduced, or proven from source with file:line evidence and a concrete failure scenario.
REFUTEDDisproving evidence found. Goes in the verdict anyway — see Stage 7.
PLAUSIBLENeither. Report it as such; never dress it as confirmed.

Worked example: finding G2-F1 (PR #4)

Reading Lib/Jobs.cs, the download worker disposes its CancellationTokenSource in a finally block, while Jobs.Cancel(id) still calls .Cancel() on job.Cancellation. Hypothesis: a cancel arriving after the download completes hits a disposed CTS and throws.

That reasoning is not a finding. It became one only after this ran in the review worktree:

[Fact]
public async Task Cancel_AfterJobCompleted_MustNotThrow()
{
    var job = Jobs.StartRealDownload(release, dir, DownloadHandler.Bytes(new byte[64]));
    await job.Completion!;
    Assert.Equal("done", job.Status);
    Assert.Null(Record.Exception(() => Jobs.Cancel(job.Id)));   // ← fails today
}
Actual: System.ObjectDisposedException: The CancellationTokenSource has been disposed.
Failed! - Failed: 1, Passed: 0

The user-facing scenario: click Cancel in the same instant the download finishes → unhandled exception in the route → HTTP 500. Confirmed.

Then delete it, and ship it

The repro test is removed from the review worktree (the reviewer commits nothing to the builder's branch) and instead quoted verbatim inside the punch list, so the builder adopts it as their own red test. The reviewer supplies the failing test; the builder makes it pass. Nobody has to trust anybody.

Stage 6 · The verdict document

One structure every time, so a reader can find any part of it without hunting. Sections in this order:

SectionContents
HeaderScope reviewed (exact diff range), method, fence check (§4), pre-flight gate result.
FindingsMost severe first. Each: one-line defect · file:line · concrete failure scenario · AC affected · verdict.
AC verificationPer AC: PASS / PARTIAL / FAIL, each answering the Stage-4 production-wiring question explicitly.
WinsGood moves named and ratified. A review that only ever lists defects trains a builder to fear the reviewer, not to learn from it.
Verified non-issuesSuspicions investigated and cleared, with the disproving evidence. This is a fence — see below.
Deviation rulingsAnything done differently from the register: ACCEPT (record an amendment) or REJECT (fix required). Never left unruled.
Punch listThree tiers. Stage 7.

Writing wins and verified non-issues is not politeness. Both carry information the builder needs: what to keep doing, and what has already been checked so they don't spend a fix round re-litigating it.

Stage 7 · Feedback to the builder session

The builder is cold — it has no memory of the review conversation and cannot read the architect's reasoning. Everything it needs must travel through artifacts it can fetch. Four mechanisms carry it.

7.1 · The three-tier punch list

Posted as a PR comment (gh pr comment 4 --body-file …), never as an in-repo report file — findings are conversation, not source. The builder reads them back with gh pr view 4 --comments.

TierMeaning to the builder
Required (block merge)Fix these, on this branch, before merge. Each is CONFIRMED with a repro.
Optional cleanupsReal but not blocking. Do not fix now — they go to the candidates list (§7.4) and get scheduled deliberately.
Verified non-issuesA fence. These were checked and cleared. Do not touch them, do not re-argue them.

The third tier does real work. Without it a builder receiving a punch list tends to "improve" adjacent code defensively, expanding the diff and the review surface. Naming what was cleared closes that door.

7.2 · The paste-able fix prompt

The architect cannot address the builder's terminal directly, so it hands the owner one line to paste. The real GAP-02 fix prompt:

Read the review comment on PR #4 (`gh pr view 4 --comments`). Implement required
finding G2-F1 on the same branch — adopt the reviewer's repro test as your red,
fix, one commit prefixed `F1:` — re-run the gates, push, post the
finding→commit→test checklist comment, and STOP. Do not touch the "verified
non-issues" items. HALT via AskUserQuestion on any ambiguity.

Its rules are load-bearing:

  • Same branch, not a new one — pushes update the existing PR in place.
  • One commit per finding, message prefixed F<n>:, so the delta re-review can isolate each fix.
  • Adopt the reviewer's repro as the red test — the fix is TDD like everything else.
  • Close with a checklist comment mapping each finding → its commits → its tests.
  • STOP again. No continuing to the next gap on a merge that hasn't happened.

7.3 · The delta re-review

When the fix lands, review only the fix-commit range — not the whole PR again, and never the builder's own checklist as evidence:

git rev-parse --short origin/feat/gap-02-real-download   # → c50cd93
gh pr checks 4                                            # green on THAT sha
git show c50cd93 --stat                                   # 2 files: guard + adopted test
git diff --name-only 31967aa..c50cd93                     # nothing else snuck in
Two conditions before upgrading to "merge it"

reviewed-commit == branch-head, and CI green on that exact SHA. A review of a commit that is no longer the head reviews nothing.

The re-review also rules on the fix itself. G2-F1 offered two options — swallow the ObjectDisposedException, or null the field before disposing. The builder chose the swallow-guard and justified it: nulling merely narrows the race window, catching closes it. The reviewer ratified that reasoning explicitly rather than silently accepting the diff.

7.4 · The candidates list

Optional cleanups do not evaporate. They accumulate in project memory (gaps-register-candidates.md), and authoring the next register starts from that list: every item must be landed, deferred with a named destination, or dropped with a written reason. Nothing silently disappears.

The live ledger holds 9 items across the four reviews. One example of the loop closing: the /api/package error handler leaked absolute filesystem paths to the client (noted on PR #3, optional tier). It was not fixed then. It was fixed by GAP-02, because that gap touched the route anyway — and the review of PR #4 marked the candidate landed.

Stage 8 · Report to the owner

Every report to the owner carries an explicit verdict line, in the owner's words, not the reviewer's hedge:

  • "merge it" — everything verified; the owner can merge without reading the diff.
  • "hold: fixes requested" — with the paste-able fix prompt attached, ready to send to the builder.

Nothing in between. "Looks mostly good" hands the owner a decision the reviewer was supposed to make.

An alarm is also a claim

Bad news gets the same evidence discipline as good news. Before telling the owner something is broken, verify it independently — a false alarm costs a round-trip and the owner's trust in every subsequent verdict.

Read the ledger honestly

One finding in four reviews is not evidence the review is unnecessary. It is one HTTP 500 that would have shipped, caught by a technique — write the failing test — that costs about ten minutes. Three approvals with zero findings are also the review working: they are what licences the owner to merge without reading 600 lines of diff.

Reviewer's checklist

Copy-paste sequence for the next review. Substitute the PR number and branch.

# 1 — CI triage FIRST. Red CI ends the review here.
gh pr checks <N>
gh pr view <N> --json commits --jq '.commits[] | .oid[0:7] + " " + .messageHeadline'

# 2 — Isolated worktree at the PR head
git fetch origin <branch>
git worktree add ../nvcleanclone-review<N> origin/<branch>
git -C ../nvcleanclone-review<N> diff --stat origin/main...HEAD

# 3 — Re-run the gates yourself (never trust the PR body)
cd ../nvcleanclone-review<N>
dotnet build nvcleanstall/CleanDriver.csproj -c Release
dotnet test  nvcleanstall/CleanDriver.Tests/CleanDriver.Tests.csproj
git diff --check origin/main...HEAD

# 3b — Exercise the changed path live
cd nvcleanstall/bin/Release/net10.0-windows && ./CleanDriver.exe --headless &
curl -s http://localhost:4780/api/<changed-route>
taskkill.exe /F /IM CleanDriver.exe

# 4/5 — Finder angles, then reproduce every suspicion as a failing test
#        (write it in the review worktree; delete it; quote it in the punch list)

# 6/7 — Post the verdict + three-tier punch list as a PR COMMENT
gh pr comment <N> --body-file verdict.md

# 8 — Clean up. Never merge.
git worktree remove ../nvcleanclone-review<N> --force

Pre-flight questions for every verdict

  • Is CI green on the exact SHA I reviewed, and is that SHA the branch head?
  • For each AC: does its test exercise the path production runs, or a lookalike?
  • Is every finding CONFIRMED with a repro or file:line proof — or honestly labelled PLAUSIBLE?
  • Did I name what was done well, and what I checked and cleared?
  • Is every deviation from the register explicitly ruled ACCEPT or REJECT?
  • Did every optional cleanup reach the candidates list, so it cannot vanish?
  • Does my report to the owner carry a bare verdict line — "merge it" or "hold"?
  • Did I remove the review worktree, and refrain from merging?
